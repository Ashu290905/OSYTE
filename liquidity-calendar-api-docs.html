Single source of truth for instrument-level key dates (notification, trade, valuation, settlement), computed on demand and published as a forward-looking calendar driven by pluggable market holiday data.

API surfaces

Surface Purpose Consumer
Compute Real-time, stateless date derivation for a single instrument + anchor date Investment Planning, Rebalancing, Trade Ops (inline)
Calendar Persisted forward-looking calendars, versioned, paginated All downstream systems, reporting, export
Changelog Audit trail of every date that moved and why Ops review, reconciliation, compliance
Holiday Data Manage base and overlay holiday calendars per tenant Ops / data-management tooling
Subscriptions Webhook delivery of calendar change events External integrations

Date computation guarantee

Date arithmetic is always deterministic — AI is never in the computation path. The Compute API and the persisted Calendar use the same engine kernel, so dates are guaranteed identical for the same instrument_id, anchor_date, and holiday_data_version. If they ever disagree, that is a defect.

Pagination

List endpoints support offset-based pagination by default (suitable for most use-cases). Endpoints that may return very large result sets (changelog, calendar entries) additionally support cursor-based pagination via a next_cursor token in the response. Prefer cursors for stable, high-volume iteration.

Rate limiting

All endpoints enforce per-tenant rate limits. Limits are communicated via response headers on every request:

Header Description
RateLimit-Limit Maximum requests allowed in the current window
RateLimit-Remaining Requests remaining in the current window
RateLimit-Reset Unix timestamp when the window resets
Retry-After Seconds to wait (present only on 429 responses)

Default limits: 600 req/min per tenant for read endpoints, 60 req/min for write endpoints. Contact Osyte to adjust.

Tracing

Every response includes X-Request-Id (UUID). Include this value when raising support requests or filing bug reports.

Versioning

Breaking changes increment the URL version (/v2/…). Additive changes (new optional fields, new endpoints) are made in-place and noted in the changelog section of this document.

Authentication

All endpoints require a Bearer JWT issued by the Osyte auth service. Tokens carry a tenant_id claim; the service enforces tenant isolation at the data layer — a token from tenant A cannot read tenant B's data.

MCP clients present the same token format.

Available scopes:

  • calendar:compute — call the Compute API
  • calendar:read — read calendars, versions, changelog
  • holiday:read — read holiday calendars
  • holiday:write — upload and manage holiday calendars
  • instrument:read — read liquidity terms
  • instrument:write — create and update liquidity terms
  • subscription:read — list subscriptions
  • subscription:write — create and delete subscriptions

Compute

Real-time, stateless date derivation. Nothing is persisted. Always reads the latest (or a specified) holiday-data version.

Compute key dates for an instrument

Returns the full set of derived key dates — notification deadline, trade date, valuation date, settlement date — for every dealing window within the requested horizon.

Stateless: nothing is persisted. Call this inline wherever you need a current, live date calculation (Investment Planning, Rebalancing, Trade Ops ticket creation).

Input options (mutually exclusive):

  • instrument_id — for instruments already onboarded in Osyte. The service loads the current effective terms automatically.
  • terms — inline liquidity terms for what-if or pre-onboarding scenarios.

Consistency guarantee: for the same instrument_id, anchor_date, and holiday_data_version, this endpoint returns identical dates to the persisted Calendar. If they disagree, that is a defect — report via X-Request-Id.

Calendar

Persisted forward-looking calendars, one per instrument. Supports current view, historical version retrieval, and async export.

Get the current forward calendar for an instrument

Returns the persisted, materialised calendar for an instrument.

Defaults to version=current — the snapshot produced by the most recent recomputation against the most recently approved holiday data.

Pass version={versionId} (from GET /calendars/{id}/versions) to retrieve a historical snapshot: "what did the calendar say on date X?" This is the primary audit and reconciliation path.

Caching: responses carry an ETag. Pass it in If-None-Match on subsequent requests — a 304 means the calendar has not changed.

Changelog

Immutable audit trail of every recomputation that produced date changes. Each entry lists affected instruments and the exact dates that moved.

Query the calendar changelog

Returns a reverse-chronological list of changelog entries — one entry per recomputation that produced at least one date change.

Each list item includes a summary count of affected instruments and changed entries. Fetch GET /changelog/{changeId} for the full diffs array.

Common use cases:

  • Ops review after a Copp Clark update: filter by holiday_source_ref
  • Compliance audit: filter by instrument_id over a date range
  • Reconciliation: cross-check EntryDiff values against OMS records

For large time ranges, use cursor-based pagination via the next_cursor field to avoid skipped records during iteration.

Upload a client-supplied holiday calendar or overlay

Accepts CSV, Excel (.xlsx), or iCal (.ics) files.

After normalisation and validation, the calendar is stored as a new immutable version. If base_calendar_id is provided, this is treated as a client overlay — merged with the base, client dates win on conflict, and every conflict is logged in the changelog.

Successful ingestion triggers automatic recomputation of all instruments whose calendar_refs include the affected market_code. The response includes an estimated instrument count; the exact count and per-instrument status are available in the resulting CalendarChanged event.

File requirements:

  • CSV: must include a date column (ISO 8601 or common formats), optional name column.
  • Excel: first sheet, same column expectations as CSV.
  • iCal: standard VEVENT components; DTSTART used as the holiday date.
  • Max file size: 10 MB.

Create a webhook subscription

Subscribe to calendar events for a set of instruments or all instruments in the tenant.

The registered url receives POST requests with:

  • Content-Type: application/json
  • X-Osyte-Signature: sha256={HMAC-SHA256 of payload body}
  • X-Osyte-Event: {event_type}
  • X-Request-Id: {delivery attempt ID}

Verify the signature using the signing_secret returned at creation (shown once; store it securely). Reject deliveries that fail verification.

Retry policy: exponential back-off, up to 5 attempts over ~1 hour. A 2xx response from your endpoint confirms delivery. Non-2xx or timeouts (>30 s) are treated as failure and retried.

See WebhookPayload schema for the delivery body structure.

Create or update liquidity terms for an instrument

Sets the liquidity terms for this instrument.

Versioning behaviour: if active terms already exist, they are closed (effective_to set to effective_from - 1 day) and a new record is created, preserving the full history. Terms can never be deleted, only superseded.

Saving new terms immediately triggers recomputation of this instrument's calendar. The response includes the saved terms record; subscribe to calendar.recomputed events or poll GET /calendars/{id} to check when recomputation is complete.
